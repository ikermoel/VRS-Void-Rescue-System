JEPA-VRS: VOID RESCUE SYSTEM — IMPLEMENTATION SPEC
====================================================

GOAL
----
Implement a causal language model for the OpenAI Parameter Golf challenge.
Constraint: artifact <= 16 MB, scored by bits-per-byte (BPB) on FineWeb.
Tokenizer: sp1024 (vocab size 1024).
Evaluation hardware: 8×H100 SXM.
Training time: 10 minutes max.


CORE IDEA
---------
Standard transformers force the model to predict directly in discrete token space.
This creates a hard problem: the model must simultaneously learn context geometry
AND the arbitrary mapping to a 1k-word vocabulary.

JEPA-VRS decouples these two responsibilities:

  Model A (The Navigator):
    - Standard causal transformer with a REGRESSION head instead of classification
    - Predicts a continuous vector in embedding space — no vocabulary knowledge
    - Primary loss: JEPA-style — L2 distance to the TARGET EMBEDDING of the real next token
    - Auxiliary loss: CE via soft BPB (to keep submission competitive)
    - Learns pure contextual geometry while remaining evaluable.

  Model B (The Rescuer):
    - A small MLP that lives on top of Model A
    - Takes the continuous vector from Model A as input
    - Learns to MOVE that vector from the "void" toward real token embeddings
    - Loss: L2 distance between its output and the real token embedding
    - Learns the topology of the embedding space — where the tokens actually live
    - Crucially: learns this INDEPENDENTLY of context (context is Model A's job)

  Inference:
    context → Model A → v_void → Model B → v_rescued → nearest_neighbor → token


WHY THIS IS DIFFERENT FROM A STANDARD TRANSFORMER
--------------------------------------------------
A standard transformer:
  context → transformer → logits (1024 values) → argmax → token
  The model is FORCED toward discrete tokens at every step.

JEPA-VRS:
  context → Model A → continuous vector (trained primarily with MSE, not CE)
           → Model B → rescued vector (geometric correction)
           → nearest neighbor (no parameters, pure geometry)

Model A learns what "should come next" as a geometric direction in continuous
space. Model B learns the map of where tokens live — independent of context.
This separation of responsibilities is the core research claim.


TWO-METRIC DESIGN (intentional)
--------------------------------
JEPA-VRS uses two metrics with different purposes:

  MSE (training, research metric):
    - The ONLY training loss — pure geometric learning, no CE
    - Measures whether Model A is learning real geometric structure
    - Answers: "Is the navigator pointing in the right direction?"

  BPB (evaluation, submission metric):
    - Computed at eval time via: v_rescued @ embed_table.T → softmax → CE
    - NOT used during training — only for reporting and submission
    - Directly comparable to baseline GPT

The gap between these two metrics is NOT a bug — it is the central research
question: "Does geometric training produce qualitatively different representations
than cross-entropy training, and at what BPB cost?"

The paper claims: a model trained exclusively with MSE (geometric loss) learns
a qualitatively different internal geometry than CE-trained transformers.
We report BPB honestly and do not claim to win the leaderboard.


MODEL A — THE NAVIGATOR
------------------------
Architecture: standard causal transformer (same blocks as baseline GPT)
  - Embedding(1024, d_model)
  - N × [CausalAttention (GQA, RoPE) + FFN (relu²)]
  - Final RMSNorm
  - NO LM head — output is raw d_model vector

Loss A (pure MSE — no CE):
  target   = embedding_table[real_next_token]    # shape: (batch, seq, d_model)
  v_void   = RMSNorm(transformer_output)         # shape: (batch, seq, d_model)
  loss_A   = MSELoss(v_void, target.detach())    # pure geometric training signal

IMPORTANT: target embeddings are detached — Model A learns to predict
the embedding space geometry, not to move the embeddings themselves.
No cross-entropy anywhere in training — the geometric constraint is pure.


MODEL B — THE RESCUER
----------------------
Architecture: small MLP, context-free
  Input:  v_void from Model A  (d_model,)
  Hidden: Linear(d_model → d_model * 2) → SiLU → Linear(d_model * 2 → d_model)
  Output: v_rescued  (d_model,)

Loss B (pure MSE — no CE):
  target     = embedding_table[real_next_token]   # same target as Model A
  v_rescued  = ModelB(v_void)                     # geometric correction
  loss_B     = MSELoss(v_rescued, target)         # L2 to real embedding

IMPORTANT: gradient flows through Model B → into v_void → into Model A.
Model B teaches Model A to produce vectors that are easier to rescue.
Fully differentiable — no straight-through estimator needed.


COMBINED TRAINING
-----------------
loss_total = 0.5 * loss_A + 0.5 * loss_B

No CE anywhere. Pure geometry end-to-end.

Training dynamic:
  - Model A learns: "predict a vector close to where the next token lives"
  - Model B learns: "given any void vector, move it to the nearest real token"
  - Together: a fully geometric pipeline, no vocabulary pressure during training

Log at every 1000 steps:
  - val_bpb     (eval only: v_rescued @ embed_table.T → softmax → CE, NOT used in training)
  - val_bpb_A   (eval only: v_void → soft logits, Model A alone)
  - loss_mse_A  (geometric accuracy of navigator — the real training signal)
  - loss_mse_B  (geometric accuracy of rescuer)
  - nn_accuracy (% nearest-neighbor correct, hard metric)


INFERENCE
---------
1. context → Model A → v_void    (continuous vector, RMSNorm output)
2. v_void  → Model B → v_rescued (geometrically corrected)
3. v_rescued @ embedding_table.T → softmax → token probabilities → BPB

For submission: standard model(x, y) interface returning CE loss on step 3.


WEIGHT TYING
------------
The embedding table (1024 × d_model) is used in three places:
  1. Input:  embedding_table[token_id]        → (d_model,) vector
  2. Loss:   target = embedding_table[target_id].detach()
  3. Logits: v @ embedding_table.T            → (1024,) dot products

The embedding table is shared across all three uses — same parameter.
This is NOT the same as the standard LM head weight tying (which outputs
vocab_size logits from a d_model→vocab_size linear). Here the transformer
output is already in d_model space and we use the embedding table as a
nearest-neighbor lookup basis, not as a classification head.

There is NO separate regression head linear. The transformer output after
RMSNorm IS the predicted vector directly. This saves d_model² parameters.


PARAMETER BUDGET (tune to fit 16 MB)
-------------------------------------
Model A: standard transformer
  vocab_size  = 1024
  d_model     = 512
  num_heads   = 8
  num_kv_heads= 4   (GQA)
  num_layers  = 9
  d_inner     = 1024  (mlp_mult=2)
  rope_base   = 10000
  tie_embeddings = True (embedding shared with logit lookup, NOT d_model→1024 head)

Model B: small MLP
  Linear(512 → 512) → SiLU → Linear(512 → 512)   [hidden_mult=1]
  Parameters: ≈ 2 × 512 × 512 ≈ 0.52M params
  NOTE: hidden_mult=2 (512→1024→512, 1.05M) gives better BPB but pushes
  artifact to 16.18 MB (over limit). hidden_mult=1 stays at 15.94 MB ✓

Total target: ≤ 17.6M parameters, artifact ≤ 16 MB after int8+zlib
  Confirmed: model_params=17,584,200 (navigator=17,059,912 + rescuer=524,288)
  Artifact: 15.94 MB ✓ (int8+zlib)


ABLATION MODELS
---------------
Model A_only:  transformer → v_void → soft logits → BPB (no Model B)
  Tests: can Model A alone produce good BPB with MSE+CE training?

Model AB_full: full JEPA-VRS (this proposal)
  Tests: does Model B improve BPB over Model A alone?

Baseline GPT:  standard transformer, CE only
  val_bpb ~1.20 at 10 min on 8×H100

Report: val_bpb, val_bpb_A, nn_accuracy, loss_mse_A, loss_mse_B, params, artifact_MB, ms/step


WHAT MAKES THIS ORIGINAL
-------------------------
1. First application of JEPA-style continuous prediction to causal LM under
   severe parameter constraints (16 MB, 10 min, vocab=1024)

2. The "Void Rescue" mechanism — an explicit geometric corrector that learns
   the topology of the token embedding space independently of context

3. Deliberate two-metric design: MSE for geometric learning, BPB for
   comparability — the gap between them is the research question

4. Nearest-neighbor inference without discrete bottleneck during training
   — fully differentiable end-to-end

Closest existing work:
  - JEPA (LeCun 2022): predicts in latent space, no rescue mechanism, not causal LM
  - VQ-VAE: discrete bottleneck, not differentiable without STE
  - Diffusion LMs: also continuous, but not the same decoupling
  - Standard transformers: forced discrete prediction, no geometric separation


PAPER CLAIM
-----------
"We propose JEPA-VRS, a causal language model that decouples contextual geometry
from lexical topology: a navigator transformer trained primarily with geometric
loss (MSE to target embedding) predicts the next token as a continuous vector,
while a small rescuer MLP learns to move void predictions toward real token
embeddings, enabling fully differentiable nearest-neighbor inference.
We report BPB for comparability and investigate whether geometric training
produces qualitatively different internal representations than standard CE training."


EXPECTED RESULTS
----------------
Target BPB range: 1.35 – 1.50
  - Sub 1.40: strong result, paper is very credible
  - 1.40–1.50: acceptable, paper focuses on representation quality
  - Above 1.50: need to increase CE weight (lower alpha)

Key figure: UMAP/t-SNE of v_void predictions vs standard transformer hiddens.
If the geometry is qualitatively different — that is the paper's main result.


ESCAPE HATCH
------------
If loss_mse_A is not decreasing by step 2000: the navigator is not learning.
  → Check embedding normalization, reduce LR.

If val_bpb > 1.70 at step 5000: BPB is too bad to be credible research.
  → Add CE auxiliary (alpha*MSE + (1-alpha)*CE, alpha=0.3) — accepts diluting the idea.

If nn_accuracy < 3% at step 5000: Model B is not learning geometry.
  → Drop Model B, submit Model A only with pure MSE training.

While testing we noticed that if the embedding lr was to high the model b stoped working correctly after step 400 in 8xh100


EMPIRICAL DISCOVERIES — LR STABILITY (critical)
-------------------------------------------------

1. EMBED_LR must be low (0.005, not 0.04)
   The embedding table is simultaneously the INPUT lookup AND the MSE training
   target. If embed_lr is too high, the embeddings move fast → the Rescuer
   chases a moving target → diverges after ~400 steps on 8×H100.
   Fix: embed_lr = 0.005 (10× lower than matrix_lr=0.04).

2. RESCUER_LR must be low (0.01, not 0.04)
   Same root cause as embed_lr: the Rescuer learns to map v_void → embeddings.
   If rescuer_lr is too high AND the Navigator is learning fast (reaching
   val_bpb_A ~1.84), the Rescuer overshoots and diverges.
   Fix: rescuer_lr = 0.01.
   Too low (0.005): Rescuer barely learns, val_bpb stuck near initialization.
   Too high (0.04): Rescuer diverges when Navigator gets good (~step 7000+).

3. WARMDOWN schedule must cover the full training budget
   The LR schedule uses time-based warmdown: if warmdown_ms > max_wallclock_ms,
   the LR declines from step 1 (the stable regime).
   Formula: WARMDOWN_ITERS × step_ms > MAX_WALLCLOCK_SECONDS × 1000
   - 10-min runs: WARMDOWN_ITERS=20000, wallclock=600s → 880,000ms > 600,000ms ✓
     LR starts at ~66% of max and declines linearly to 0. Rescuer stable.
   - 1-hour runs: WARMDOWN_ITERS=120000, wallclock=3600s → 5,280,000ms > 3,600,000ms ✓
     LR starts at ~67% of max — equivalent schedule to 10-min runs.
   WARNING: using WARMDOWN_ITERS=20000 with wallclock=3600s keeps LR at 100%
   for 45 min → Rescuer diverges. Always scale WARMDOWN_ITERS with wallclock.

4. ITERATIONS default must exceed total step count
   At 44ms/step on 8×H100: 10-min ≈ 13,600 steps, 1-hour ≈ 81,800 steps.
   Default ITERATIONS=100000 ensures the wallclock cap fires before step cap.
   Old default (20000) would stop a 1-hour run at ~15 min.

5. Checkpoint resume: training_time_ms must be saved and restored
   On resume, if training_time_ms resets to 0, the LR warmdown sees
   artificially low elapsed time → warmdown never triggers → LR stays high
   → Rescuer diverges in late training.
   Fix: save/restore training_time_ms in checkpoint alongside step.


CONFIRMED RESULTS (8×H100, 10 min, seeds 7 / 42 / 1337)
----------------------------------------------------------
All three seeds converge to virtually identical results:
  seed 7:    val_bpb=1.8660  val_bpb_A=1.9586  nn_acc=50.3%  step=13432
  seed 42:   val_bpb=1.8675  val_bpb_A=1.9291  nn_acc=50.5%  step=13469
  seed 1337: val_bpb=1.8680  val_bpb_A=1.9410  nn_acc=50.7%  step=13430
  artifact:  15.94 MB (under 16 MB limit ✓)
  ms/step:   44.55–44.69ms

Key observation: at step ~12000 in 10-min runs, LR scale ≈ 7% (near end of
warmdown). Both Navigator and Rescuer are in final slow descent together.
The Rescuer consistently improves BPB by ~0.09 over Navigator alone.


RECOMMENDED RUN COMMANDS
--------------------------
10-minute run (baseline/submission):
  MAX_WALLCLOCK_SECONDS=600 SEED=42 torchrun --standalone --nproc_per_node=8 train_jepa_vrs.py

1-hour run (extended training):
  MAX_WALLCLOCK_SECONDS=3600 SEED=42 ITERATIONS=100000 WARMDOWN_ITERS=120000 RESCUER_LR=0.01 \
  torchrun --standalone --nproc_per_node=8 train_jepa_vrs.py

Continue from checkpoint (same total budget):
  MAX_WALLCLOCK_SECONDS=<total_seconds> CHECKPOINT_PATH=./checkpoint_jepa_vrs.pt \
  [same other vars] torchrun --standalone --nproc_per_node=8 train_jepa_vrs.py
  NOTE: MAX_WALLCLOCK_SECONDS must be TOTAL elapsed target, not additional time.









Me gusta — es elegante y no requiere lógica adaptativa compleja. Es simplemente un scheduler más agresivo para el Rescuer.
La intuición es correcta: el Rescuer necesita aprender la geometría básica rápido al inicio (LR alto) y luego estabilizarse antes de que el Navigator se aleje demasiado (decay agresivo).
El análisis de Claude Code es sólido:
Step 200:  Rescuer LR = 0.036  — aprende geometría básica rápido
Step 6800: Rescuer LR = 0.010  — estable, Navigator no lo deja atrás
Step 20000: Rescuer LR = ~0.005 — casi fijo, solo refinamiento
Eso es exactamente el comportamiento que necesitabas para el run de 1 hora.
Para el paper esta decisión de diseño es un hallazgo en sí mismo — no es solo un trick de entrenamiento, es una observación sobre la dinámica de co-entrenamiento de dos módulos con responsabilidades distintas:
"El Rescuer requiere un warmdown más agresivo que el Navigator para mantener co-evolución estable. Esto refleja la asimetría fundamental entre aprender geometría contextual (Navigator, tarea continua) y aprender topología de embeddings (Rescuer, tarea que converge más rápido)."
Deja correr el run y mándame los números. Si llega a step 13,000+ sin divergencia, tienes el resultado del run de 1 hora que faltaba.

"La divergencia observada en runs extendidos no era un problema arquitectural de JEPA-VRS — era un problema de scheduler. El LR del Rescuer permanecía demasiado alto en late training cuando el warmdown estaba calibrado para runs largos. La solución: aplicar el mismo warmdown de los runs de 10 minutos y continuar con LR_MIN_SCALE=0.02. Con este fix, JEPA-VRS es estable indefinidamente."
